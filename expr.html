Requires expressions

A requires-expression provides a concise way to express syntactic requirements on template arguments. A syntactic requirement is one that can be checked by name lookup () or by checking properties of types and expressions. A requires-expression defines a conjunction of constraints (), which are introduced by its requirements. requires-expression requires requirement-parameter-list requirement-body requirement-parameter-list ( parameter-declaration-clause ) requirement-body { requirement-list } requirement-list requirement ; requirement-list requirement requirement simple-requirement type-requirement compound-requirement nested-requirement

A requires-expression has type bool.

A requires-expression evaluates to true when all of its introduced constraints are satisfied (), and false otherwise.

A requires-expression shall appear only within a concept definition () or a requires-clause following a template-parameter-list (), init-declarator (), function-definition (), or member-declarator (). The most common use of requires-expressions is to define syntactic requirements in concepts such as the one below: template<typename T> concept bool R() { return requires (T i) { typename A<T>; {*i} -> const A<T>&; }; } A requires-expression can also be used in a requires-clause templates as a way of writing ad hoc constraints on template arguments such as the one below: template<typename T> requires requires (T x) { x + x; } T add(T a, T b) { return a + b; } The first requires introduces the requires-clause, and the second introduces the requires-expression.

A requires-expression may introduce local parameters using a parameter-declaration-clause (). Each name introduced by a local parameter is in scope from the point of its declaration until the closing brace of the requirement-body. These parameters have no linkage, storage, or lifetime; they are only used as notation for the purpose of defining requirements.

A local parameter shall not have a default argument.

The requirement-parameter-list shall not include an ellipsis.

The requirement-body defines a block scope, and is comprised of a sequence of requirements. These requirements may refer to local parameters, template parameters, and any other declarations visible from the enclosing context. Each requirement appends one or more atomic constraints () to the conjunction of constraints defined by the requires-expressions.

Simple requirements

simple-requirement expression

A simple-requirement introduces an expression constraint () for its expression. template<typename T> concept bool C = requires (T a, T b) { a + b; // a simple requirement };

The substitution of template arguments into the expression of a simple-requirement may result in an ill-formed expression. However, the resulting program is not ill-formed.

If the expression would always result in a substitution failure, the program is ill-formed. template<typename T> concept bool C = requires () { new T[-1]; // error: the expression will never be well-formed };

Type requirements

type-requirement typename-specifier

A type-requirement introduces type constraint () for the type named by its typename-specifier. A type requirement asserts the validity of an associated type, either as a member type, a class template specialization, or an alias template. It is not used to specify requirements for arbitrary type-specifiers. template<typename T> struct S { }; template<typename T> using Ref = T&; template<typename T> concept bool C = requires () { typename T::inner; // required nested type name template S<T>; // required class template specialization typename Ref<T>; // required alias template substitution };

The substitution of template arguments into the typename-specifier of a type-requirement may result in an ill-formed type. However, the resulting program is not ill-formed.

If the required type would always result in a substitution failure, then the program is ill-formed. template<typename T> concept bool C = requires () { typename int::X; // error: int does not have class type };

Compound requirements

compound-requirement constexpr { expression } noexcept trailing-return-type ;

A compound-requirement introduces a conjunction of one or more atomic constraints related to its expression. A compound-requirement appends an expression constraint () for its expression, E. The constexpr specifier, if present, appends a constant expression constraint () for E. The noexcept specifier, if present, appends an exception constraint () for E. If the trailing-return-type is given, and its type (call it T), contains no placeholders (, ), the requirement appends two constraints to the conjunction of constraints: a type constraint on the formation of T () and an implicit conversion constraint from E to T (). Otherwise, if T contains placeholders, an argument deduction constraint () of E against the type T is appended to the conjunction of constraints. In this constraint, deduction is performed against an invented function template whose declaration is formed according to the rules for abbreviated functions in where T is the type of the invented function's only parameter.

template<typename I> concept bool C() { return true; } template<typename D> concept bool D = requires(T x) { {x++}; // #1 {*x} -> typename T::inner; // #2 {f(x)} -> const C&; // #3 {g(x)} noexcept; // #4 constexpr {T::value}; // #5 }; Each of these requirements introduces a valid expression constraint on the expression in its enclosing braces. Requirement #1 only an expression constraint on x++. It is equivalent to a simple-requirement with the same expression. Requirement #2 *x introduces three constraints: an expression constraint for *x, a type constraint for typename T::inner, and a conversion constraint requiring *x to be implicitly convertible to typename T::inner. Requirement #3 also introduces two constraints: an expression constraint for f(x), and a deduction constraint requiring that overload resolution succeeds for the call i(f(x)) where i is the following invented function template. template<typename T> requires C<T>() void i(const T&); Requirement #4 introduces two constraints: an expression constraint for g(x) and an exception constraint on g(x). Requirement #5 also introduces two constraints: An expression constraint on T::value and a constant expression constraint that T::value.

The substitution of template arguments into the expression of a simple-requirement may result in an ill-formed expression. However, the resulting program is not ill-formed.

Nested requirements

nested-requirement requires-clause ;

A nested-requirement can be used to specify additional constraints on a type declared by a type-requirement (). It appends a normalization of the constraint-expression of its requires-clause to the conjunction of constraints (). template<typename T> concept bool C() { return sizeof(T) == 1; } template<typename T> concept bool D = requires () { typename X<T>; requires C<X<T>>(); }; The nested-requirement appends the predicate constraint sizeof(T) == 1 ().
